# Milestone 15 — UI simplification and user profile

## Profile architecture

`GET /profile` renders the authenticated user's account and the linked Person. Two
CSRF-protected commands handle email and password changes:

- `POST /profile/email`
- `POST /profile/password`

`ProfileController` delegates validation and writes to `ProfileService`.
`UserRepository::updateEmail()` is implemented by both repository adapters.
Passwords are verified with `password_verify()` and replaced only with a
`password_hash()` generated by `AuthenticationService`; hashes are never sent to
the view.

The page shows username and account role as read-only values. Email and password
are editable. A linked Person's name, surname, position, affiliation, and annual
capacity are read-only. The current Person schema has no phone or office fields.
Its `notes` field is administrative, so it is intentionally not exposed as a
self-service field. Missing Person links produce a warning and remain managed in
Users administration.

All authenticated roles may access their own profile. Existing administration,
People management, and authorization policies are unchanged.

## UI and effort cleanup

- The top-right account control is now a keyboard-accessible Bootstrap menu with
  My Profile, role-gated Administration, and Logout.
- Capacity search and actions use separate flex groups. Expand/Collapse buttons
  are content-sized (`flex: 0 0 auto; width: auto`) on desktop and mobile.
- Capacity, allocation, participant, project, Work Package, and annual effort
  views now present one `Allocated hours` value, one remaining-capacity value,
  and one derived PM value.
- Allocation create/edit accepts one `allocated_hours` input. The service writes
  the same value through the existing persistence compatibility layer.
- Old inconsistent records are still identified as legacy data and remain
  read-only until reconciled. New UI writes cannot create divergent values.

The database columns and repository aliases named `planned_hours` and
`actual_hours` remain temporarily as a storage compatibility layer for existing
installations and historical tests. They are no longer separate user concepts;
new writes keep them identical and models expose `allocatedHours()`. Physically
dropping these columns requires a dedicated data migration after every deployed
database has reconciled divergent historical rows.
